# Fact-SSOT Meta-Rule (Public Edition)

> **Rule level**: Synapse system-level meta-rule, equal in priority to CEO Guard and the 350-line CLAUDE.md entropy budget.
> **Triggers**: Any document, web page, blog post, report, Slack message, or API response that mentions a concrete factual number.
> **Applies to**: All Agents, all Workers, all human contributors.
> **Enforcement**: CI lint + weekly audit + Lysander review on violation.

---

## 1. Why this rule exists

In April 2026 the same fact — "number of Agents in the Synapse system" — was found stated three different ways across the public website: **44**, **46**, and **50**. The actual value (computed from `modules/*/agents/*.md` in this repo) was **53 unique / 54 files / 13 modules**.

The root cause was not a typo. There was simply **no single source of truth**: each page, blog, and report froze a different number at a different moment. This rule prevents that class of drift from recurring.

---

## 2. Rule statement

> **Every "fact-type number" must be sourced from a single source of truth (SSOT).
> Documentation, web pages, blogs, reports, Slack notifications, and API responses
> MUST NOT hardcode any fact-type number.**

### 2.1 What counts as a "fact-type number"

Fact-type number = a countable value that describes the **current real state** of the system or product.

**MUST be SSOT-sourced**:
- Agent count (e.g. 53, 54)
- Module / team count (e.g. 13)
- Version number (e.g. v2.1.0)
- Line-budget caps (e.g. CLAUDE.md ≤ 350 lines)
- Pricing (e.g. training $500)
- Audit score thresholds (e.g. ≥ 85/100)
- Token budgets (e.g. 200k context)
- Customer count / project count / shipped count

**NOT subject to this rule**:
- Approximations in prose ("dozens", "hundreds") — but never co-located with a precise figure
- Decorative mockups (e.g. fake CPU% in a terminal animation) — must be clearly labelled as mockup
- Mathematical / physical constants (π, c)
- Historical snapshots that explicitly state "as of YYYY-MM-DD"

### 2.2 Scope

**Repos and surfaces required to comply**:
- `synapse-core` — this repo (`docs/public/`, `CLAUDE.md`, `README.md`)
- `lysander-bond` — public website (`src/pages/`, `src/content/blog/`, `src/components/`)
- Internal working repos (e.g. `Synapse-Mini`)
- All auto-generated reports (intelligence dailies, action reports, PMO weekly)
- Slack notification templates
- All published demo / marketing materials

**Exceptions**:
- Quoting a user's input ("the user said 50") — quotation, not assertion
- Archived historical documents explicitly marked `frozen at YYYY-MM-DD`

## 3. The SSOT file

**Path**: `docs/public/synapse-stats.yaml` (this repo)

Auto-generated by `scripts/generate-stats.mjs`. Schema:

```yaml
version: "<reads VERSION>"
generated_at: "<ISO-8601>"

agents:
  total_unique: <int>   # unique Agent IDs across all modules
  total_files:  <int>   # all .md files under modules/*/agents/
  by_module:
    <module_id>: <int>
    ...

modules:
  total: <int>
  list: [<sorted module ids>]

presets:
  total: <int>
```

Regenerate on demand:
```bash
node scripts/generate-stats.mjs
```

The script is idempotent — re-running produces identical output (modulo `generated_at`).

## 4. How downstream consumers comply

### 4.1 Build-time injection (preferred)

Build pipelines pull `synapse-stats.yaml` into the project at prebuild time and reference fields rather than hardcoding numbers.

**Example: `lysander-bond`**
```js
// scripts/sync_from_core.mjs runs before astro build
// Pulls synapse-core/docs/public/synapse-stats.yaml → src/data/synapse-core/synapse-stats.yaml

// In a page or component:
import stats from '../data/synapse-core/synapse-stats.yaml';
// → {stats.agents.total_unique} renders the live number
```

### 4.2 Manual writing (blogs, reports)

If build-time injection is impractical (e.g. a long-form blog post), the document MUST include a footer line:

> *Data snapshot: YYYY-MM-DD (synapse-core@&lt;commit-sha&gt;)*

This converts a "current fact" claim into a "historical snapshot" claim, which is permitted under §2.1.

### 4.3 CI enforcement (recommended)

Downstream repos SHOULD run a lint that scans for hardcoded fact-type number patterns and fails the build when matches do not equal the SSOT value. Reference implementation: `lysander-bond/scripts/lint-no-hardcoded-stats.mjs`.

---

## 5. Defence in depth (three layers)

| Layer | Mechanism | When it fires |
|-------|-----------|---------------|
| **L1 Build-time injection** | `synapse-stats.yaml` + lint-no-hardcoded-stats | Every push to consumer repos |
| **L2 Weekly audit** | `scripts/audit_facts.py` cron | Every Monday 00:00 UTC, scans all repos |
| **L3 Authoring hint** | Skill / hook prompts when writing numbers | At the moment of human authorship |

A violation that escapes all three layers and is found by the President is treated as a system failure and triggers a root-cause analysis.

## 9. Five-line cheatsheet

> 1. Any concrete fact-type number (Agent count, team count, version, price, line cap) → read from SSOT, never hardcode.
> 2. SSOT file: `synapse-core/docs/public/synapse-stats.yaml`, auto-regenerated on every push.
> 3. Web pages, blogs, reports that mention numbers → import stats, or footer `Data snapshot YYYY-MM-DD`.
> 4. CI scans for hardcoded numbers; matches that disagree with SSOT fail the build. No exceptions.
> 5. The President should never be the first to spot a number drift — that is a system failure.
